## 附：派遣制工资表的结构特点

> 当前处理的是**人才派遣（劳务派遣）**员工工资表，其结构不同于常规在职员工工资表：

| 特点 | 说明 |
|------|------|
| **收入项少** | 通常只有基本工资、补发工资等少数几个收入子项，没有岗位津贴、绩效、交通补贴等复杂子项 |
| **五险个人打进应发** | 因为收入子项太少，部分单位的系统把**五险一金个人扣款**直接打到应发工资里作为组成部分 |
| **扣款明细再扣回** | 五险一金个人部分在扣款明细中再次列出扣除，加一次减一次，**两者抵消** |
| **净效果简单** | 员工实际到手 ≈ 基本工资 + 补发工资 − 单位代理费 − 个人所得税 |

### 为什么会这样

派遣制员工的薪酬结构简单，原始工资系统在设计应发工资的组成时，把五险一金个人部分也纳入应发工资总额，再在扣款明细中扣除。这在会计处理上是"列示方式"问题，实际不产生重复扣款。

**直观理解**：
```
应发工资（总包） = 基本工资 + 五险一金个人部分
扣款（明细列示） = 五险一金个人部分 + 单位代理费 + ...
实发（到手）     = 应发 - 扣款 - 个税
                  = 基本工资 - 单位代理费 - 个税
```

这就是为什么文档中**应发工资的精确公式不做死校验**的原因——各单位系统的处理方式不同，但资金流向恒等式始终成立。

## 二、数据关系验证（已通过实际数据验证）

> 以下公式均使用 2026年06月 实际工资表数据验证通过。

### 公式 1：应发工资的构成因单位而异

**验证结果**：📌 **不做统一公式校验**

**说明**：
- 不同单位的原始工资表列结构不同，导致应发工资的构成也不同
- 当前数据存在两种情况：

| 情况 | 示例 | 规律 |
|------|------|------|
| **应发工资 = 基本工资** | 王单怡 11364.82 = 11364.82 | 仅有基本工资 |
| **应发工资 = 基本工资 + 补发工资** | — | 含补发工资项 |
| **应发工资 = 基本工资 + 五险一金个人部分** | 仲佳奇 7123.99 = 6257.06 + 866.93 | 五险个人打进应发 |
| **应发工资 = 基本工资 + 补发工资 + 五险一金个人部分** | 李志勇 7756.86 = 6546.39 + 601.97 + 608.50 | 兼有 |

- **五险一金个人部分**: 在扣款明细中列出（养老个人+失业个人+医疗个人+公积金个人），单位不同处理不同
- 应发工资的精确公式取决于原始工资表的列定义，不应用单一公式强制校验

**建议**：仅校验资金流向恒等式（公式4），不对应发工资的构成做死公式校验

---

### 公式 2：转账合计 = 应发工资 + 单位缴纳五险一金 + 单位代理费 + 雇主责任险

**验证结果**：✅ **完全成立**（合计行和所有数据行均已验证）

```
327849.95 = 266095.69 + 60354.26 + 1400.00 + 0.00 ✅
```

逐行验证（前3行）：
| 姓名 | 应发工资 | +五险一金 | +代理费 | +雇主险 | =转账合计 |
|------|---------|----------|--------|--------|----------|
| 王单怡 | 11364.82 | 3595.18 | 40.00 | 0.00 | 15000.00 ✅ |
| 李晓虹 | 5500.00 | 0.00 | 40.00 | 0.00 | 5540.00 ✅ |
| 马雪梅 | 6000.00 | 0.00 | 40.00 | 0.00 | 6040.00 ✅ |

**校验规则**：
```python
assert abs(转账合计 - (应发工资 + 单位缴纳五险一金 + 单位代理费 + 雇主责任险)) <= tolerance
```

### 公式 4：转账合计 = 扣款合计 + 个人所得税 + 实发合计

**验证结果**：✅ **完全成立**

```
327849.95 = 92342.50 + 2147.63 + 233359.82 ✅
```

这实质上是工资表的**资金流向恒等式**：
```
进入（转账合计）= 中间扣留（扣款合计 + 个税）+ 最终到手（实发合计）
```

---

### 公式 5：实发工资 == 实发合计（当前数据，无个人调整列）

**验证结果**：✅ 两者完全相等

```
实发工资(233359.82) == 实发合计(233359.82) ✅
```

**说明**：当前批次的工资表没有"个人其他调整"列，所以实发工资和实发合计的值相同。
如果其他批次的数据源包含"个人其他调整"列，关系会变为：
- **实发工资(扣调整前) = 实发合计(最终值) + 个人其他调整 + 其他扣除项**
- 即实发工资包含实发合计和各种调整项，调整项从实发工资中扣除后得到实发合计

**校验规则**（针对当前数据）：
```python
# 无个人调整列，直接验证相等
assert abs(实发合计 - 实发工资) <= tolerance
```

## 三、从 submit_approval_feishu 借鉴的校验方法

### A. 纵向列加总校验（column_sum_checks）

**原理**：对某列累加所有数据行，与合计行该列的值比较。

**建议应用到 seqprint 的列**：
```json
"column_sum_checks": [
  {"column": "基本工资"},
  {"column": "补发工资"},
  {"column": "应发工资"},
  {"column": "单位缴纳五险一金"},
  {"column": "转账合计"},
  {"column": "扣款合计"},
  {"column": "个人所得税"},
  {"column": "实发工资"},
  {"column": "实发合计"}
]
```

### B. 横向公式校验（row_formulas）

**原理**：验证每行的恒等式 lhs == (sum rhs_plus) - (sum rhs_minus)。

**建议应用到 seqprint 的公式**（基于实际数据验证）：

#### 公式 1：转账合计构成
```
name: "转账合计 = 应发工资 + 单位缴纳五险一金 + 单位代理费 + 雇主责任险"
lhs: "转账合计"
rhs_plus: ["应发工资", "单位缴纳五险一金", "单位代理费", "雇主责任险"]
rhs_minus: []
```

#### 公式 2：资金流向恒等式
```
name: "转账合计 = 扣款合计 + 个人所得税 + 实发合计"
lhs: "转账合计"
rhs_plus: ["扣款合计", "个人所得税", "实发合计"]
rhs_minus: []
```

#### 公式 3：扣款合计 = 所有扣款子项（如有扣款子项明细数据）
```
name: "扣款合计 = 养老(单位+个人) + 失业(单位+个人) + 医疗(单位+个人) + 工伤险 + 公积金(单位+个人) + ..."
lhs: "扣款合计"
rhs_plus: ["养老(单位)", "养老(个人)", "失业(单位)", "失业(个人)", 
           "医疗(单位)", "医疗(个人)", "工伤险", 
           "公积金(单位)", "公积金(个人)",
           "公务员医疗补助", "单位代理费(扣款下)", 
           "大病险（单位）", "雇主责任险(扣款下)"]
rhs_minus: []
```

> **注意**：公式3要求数据行有完整的扣款子项明细（col 17-29）。如果数据来源只提供总计没有明细，则跳过此校验。

### C. 表格格式校验（table_format）

**原理**：检查必备列是否存在、合计行是否有实际值。

**seqprint 的关键列**：
```python
critical_cols = ["基本工资", "应发工资", "转账合计", "扣款合计", "实发合计"]
```

### D. 签名栏检查

**原理**：扫描全局关键字。seqprint 中签名字段在输出文件的固定位置：
```
col 1: 总经理签字：
col 7: 部长签字：
col 13: 财务审核：
col 19: 制表人：张朦
```

## 四、容差设计

| 参数 | 建议值 | 说明 |
|------|-------|------|
| `tolerance` | 0.00 | 严格模式下不允许误差。所有金额精确到分（2位小数） |
| `strict` | true | 严格模式：校验失败禁止提交/打印 |
| rounding | ROUND_DOWN | 分以下直接舍去（银行报盘处理逻辑） |

> 注意：openpyxl 读取 Excel 时可能因浮点精度引入 ±0.00000001 误差，
> 建议在 `round(val, 2)` 后再比较，或设 tolerance = 0.005。
