# 虚拟工资集合表表头生成逻辑（指纹方法）

> 对应代码：`batchprint_gui.py` → `merge_payrolls_simple()` 函数
> 提交版本：`100431c` 及之后

---

## 一、总体布局

合并输出的工资表合集（虚拟表）从上到下的结构：

```
行1: 大标题        ← {单位名} 工资表合集（{年月}），整行水平合并
行2: 单位信息      ← 左"单位名称：{单位名}" + 右"打印时间：{日期}"
行3: 三级表头-第1层 ← 序号 | 结算单元名称 | [规范指纹序的第3行值]
行4: 三级表头-第2层 ← (空) | (空)        | [规范指纹序的第4行值]
行5: 三级表头-第3层 ← (空) | (空)        | [规范指纹序的第5行值]
行6+: 每个源文件一条合计行
  合计行: 序号 | 单位名 | [各列合计值]
行N+2: (空行)
行N+3+: 源文件原样堆叠（含签名图片，通过 WPS COM Paste 复制）
```

整个虚拟表使用 WPS COM (`win32com.client.Dispatch("KET.Application")`) 构建。

---

## 二、总体流程

```
扫描文件 → 分组 → 构建指纹 → 过滤空列 → display_hdr → 写表头 → 合并 → 写数据
```

关键区别（对比旧方案——按列号映射 + 复制源合并格）：
- 旧方案：以第1个源文件的列号为主，复制其 `merged_cells.ranges` 做合并
- 新方案：以**指纹** (`(r3v, r4v, r5v)`) 为列标识，合并规则通过**扫描已写入值**动态判定

---

## 三、构建规范列指纹序列（`canonical_fps`）

指纹 = `(行3的值.strip(), 行4的值.strip(), 行5的值.strip())`，三元组唯一标识一列。

### 3.1 提取指纹

每个源文件独立生成 `fp_dict: {源列号 → (r3v, r4v, r5v)}`。合并格处理：行3-5范围内的合并区域，将左上角值扩散填充到所有被覆盖格。

```python
merge_lookup = {}
for mc in src_ws.merged_cells.ranges:
    if 行3-5范围内:
        将左上角值写入 merge_lookup[(行, 列)]

def _cv(row, col):
    if (row, col) in merge_lookup: return merge_lookup[(行, 列)]
    return str(src_ws.cell(row, col).value).strip() 或 ""

fp_dict[列号] = (_cv(3, 列号), _cv(4, 列号), _cv(5, 列号))
```

### 3.2 选参考文件

以最常见的列数字段为参考（白云97个文件中30列占73个，选30列文件为参考）。

### 3.3 构建有序序列

**Step 1**：参考文件的指纹顺序（C7起，去重）作为基础序列：
```python
for c in range(7, max_cols + 1):
    fp = ref_fp.get(c)
    if fp 非空且未见过:
        canonical_fps.append(fp)
```

**Step 2**：其他变体文件的独有指纹，**位置感知插入**而非追加到末尾：
```python
for fpf in file_fingerprints:
    for c, fp in fpf.items():
        if fp 非空且未见过:
            # 在同r3v组中找到第一个"合计"项，插到它前面
            # 若无"合计"则插到同r3v组末尾
            # 若无可匹配的r3v组则追加到尾部
```

**为什么位置感知插入**：变体独有列若追加到末尾，会落在"扣款合计"后面——而各组r3v是非连续的（不同列区块交错排列），后续无法通过组内重排把"公务员医疗补助"移到"扣款合计"前面。位置感知插入确保同类指纹在规范序列中连续。

### 3.4 过滤全空列

遍历每个指纹列在所有源文件的合计行数据，若全部为0或空值则从 `canonical_fps` 中移除（即虚拟表中不展示该列）。

## 四、生成显示表头（`display_hdr`）

`canonical_fps` 中的指纹用于**数据映射**（内部定位列身份），但虚拟表显示时需要**原始表头值**（不带指纹的简洁文字）。

### 4.1 读取参考文件的原始表头

```python
ref_hdr = [{}, {}, {}]   # ref_hdr[0]=行3, ref_hdr[1]=行4, ref_hdr[2]=行5
for hi, hr in enumerate([3, 4, 5]):
    for c in range(1, max_cols + 1):
        v = ref_ws.cell(row=hr, column=c).value
        ref_hdr[hi][c] = str(v).strip() if v else ""

# 行3向右传播：解决 openpyxl 读取合并格只有左上角有值的问题
last_val = ""
for c in range(1, max_cols + 1):
    v = ref_hdr[0].get(c, "")
    if v:
        last_val = v
    ref_hdr[0][c] = last_val
```

### 4.2 构建 display_hdr 映射

```python
display_hdr: {fingerprint: (行3显示值, 行4显示值, 行5显示值)}
```

对参考文件中每个数据列（C7起），取该列的指纹和参考文件原始表头值，建立映射：
```python
for c in range(7, max_cols + 1):
    fp = ref_fp.get(c)
    if fp and fp != ("", "", ""):
        display_hdr[fp] = (ref_hdr[0][c], ref_hdr[1][c], ref_hdr[2][c])
```

**关键设计**：`display_hdr` 只构建参考文件中**已存在**的指纹。变体独有指纹（如公务员医疗补助、大病险等）不在参考文件中，因此在 `display_hdr` 中不存在。写表头时通过 `display_hdr.get(fp, fp)` 回退到指纹元组自身的值——这也是正确的，因为指纹中的 r4v/r5v 就是源文件中的列头文字。

## 五、写入3行复合表头

先写大标题和单位信息行（行1-2），再写三级复合表头（行3-5）：

```python
r = hdr_start_row
for hi in range(3):                    # 行3(h=0), 行4(h=1), 行5(h=2)
    tgt_ws.Cells(r, 1).Value = "序号" if hi == 0 else ""
    tgt_ws.Cells(r, 2).Value = "结算单元名称" if hi == 0 else ""
    for vi, fp in enumerate(canonical_fps, 3):
        dv = display_hdr.get(fp, fp)[hi]   # 有display_hdr用原始值，否则用fp回退
        tgt_ws.Cells(r, vi).Value = dv
    r += 1
```

以 30列参考文件 + 1个31列变体（含公务员医疗补助）为例，写入后的裸数据：

```
行3: 序号 | 结算单元名称 | 基本工资 | 交通补贴 | 应发工资 | ... | 扣款明细 | 扣款明细 | 扣款明细 | 扣款明细 | ... | 个人所得税 | 实发工资 | 实发合计
行4:      |              | 基本工资 | 交通补贴 | 应发工资 | ... | 养老     | 养老     | 失业     | 公务员医疗补助 | ... | 个人所得税 | 实发工资 | 实发合计
行5:      |              | 基本工资 | 交通补贴 | 应发工资 | ... | 单位     | 个人     | 单位     | 公务员医疗补助 | ... | 个人所得税 | 实发工资 | 实发合计
```

此时各单元格是独立的，还未合并。

## 六、合并单元格

通过扫描已写入的单元格内容，将相邻且值相同的合并。**不依赖源文件的 `merged_cells.ranges`**——这个方法是指纹方法的核心差异之一。

### 6.1 行3水平合并

扫描行3，合并连续相同值的单元格：

```
扫描前: [扣款明细] [扣款明细] [扣款明细] [扣款明细] [扣款明细] ...
扫描后: [──────────────── 扣款明细 ────────────────] ...
```

条件：`vj - 1 > vi`（至少2列相同才合并）。

### 6.2 行4水平合并

在行3同组内（即同一个行3合并块的覆盖范围内），合并行4连续相同值：

```
行3: [─────────── 扣款明细 ──────────]
行4: [养老] [养老] [失业] [公务员医疗补助] [单位代理费] [扣款合计]
      └─合并─┘
```

条件：在同组内，至少2列行4值相同。

### 6.3 纵向合并

按三种情形处理：

| 情形 | 条件 | 例子 | 范围 |
|------|------|------|------|
| A | r3v==r4v==r5v | 基本工资、个人所得税、实发合计 | 行3-5（3行全合并） |
| B | 仅r4v==r5v, r3v不同 | 扣款明细区内的单位代理费、公务员医疗补助 | 行4-5（2行合并） |
| C | r4v有值, r5v为空 | 工伤险（原始格式中常见） | 行4-5（2行合并） |

```python
for vi in range(3, virtual_cols + 1):
    r3v = 行3该列的值
    r4v = 行4该列的值
    r5v = 行5该列的值
    if r4v 有值 and (r5v 为空 or r4v == r5v):
        if r3v == r4v:
            合3行（行3-5）
        else:
            合2行（行4-5）
```

**排除**：r4v != r5v 的列不合并（如"养老/单位"和"养老/个人"是两列不同的子项，各自独立）。

### 6.4 其余列：跨3行合并

对仍未参与任何合并的列，若行4+行5都为空值，合并行3-5（如"序号"列）。

```python
for vi in range(1, virtual_cols + 1):
    if 该列行3/4/5已有合并:
        continue
    if 行4为空 and 行5为空:
        合并该列行3-5
```

## 七、合并防冲突机制

使用 `hmerged` 集合（`{(行号, 列号)}`）记录已合并的单元格，防止重复合并引发 WPS COM 异常。

```
第一次合并 → 标记 (行1, 列A) 到 (行1, 列E) 到 hmerged
第二次合并 → 检查目标范围是否在 hmerged 中有记录 → 无则合并 → 有新标记
```

## 八、数据写入

每个源文件的合计行数据，通过 `{指纹 → 源列号}` 反向映射写入虚拟表对应列：

```python
rev = {fp: 源列号 for 源列号, fp in fp_dict.items()}
for vi, fp in enumerate(canonical_fps):
    源列号 = rev[fp]   # 由指纹定位到该文件的正确列
    if 该列有合计值:
        写入 tgt_ws.Cells(数据行, vi + 3)
```

最后在末尾写入合计行的汇总值。

## 十、常见问题

### Q: 为什么不用复制源文件的 merged_cells.ranges？

旧方案就是复制合并格，但有以下问题：
- 不同变体文件的列数不同，合并格位置不同
- 变体独有列的合并信息不存在于参考文件
- 空列过滤后列映射不连续，合并格无法正确映射

指纹方法不依赖具体的合并格位置，而是通过内容匹配和写入值扫描来动态合并，自适应所有变体。

### Q: 公务员医疗补助为什么会在扣款合计后面？

见 `fix-merge-column-mapping.md` 计划。原因为变体独有指纹追加到 `canonical_fps` 末尾，而各组 r3v 在规范序列中非连续（被个人所得税、实发工资等穿插），导致组内重排无法跨段修复。修复方式：位置感知插入——变体指纹插入到同 r3v 组第一个"合计"项之前。
