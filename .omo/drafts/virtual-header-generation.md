# 虚拟工资集合表表头生成逻辑

> 对应代码：`batchprint_gui.py` → `merge_payrolls_simple()` 函数
> 核心区域：第 529~647 行

---

## 一、总体布局

合并输出的工资表合集（虚拟表）从上到下的结构：

```
行1: 大标题        ← {单位名} 工资表合集（{年月}），整行水平合并
行2: 单位信息      ← 左"单位名称：{单位名}" + 右"打印时间：{日期}"
行3: (空行)
行4: 三级表头-第1层 ← 序号 | 结算单元名称 | [源文件行3的列头]
行5: 三级表头-第2层 ← (空) | (空)        | [源文件行4的子列头]
行6: 三级表头-第3层 ← (空) | (空)        | [源文件行5的子列头2]
行7+: 每个源文件一条合计行
  合计行: 合计 | [源文件名] | [各列合计值]
行N+2: (空行)
行N+3+: 源文件原样堆叠（含签名图片，通过 WPS COM Paste 复制）
```

整个虚拟表使用 WPS COM (`win32com.client.Dispatch("KET.Application")`) 构建。

## 二、三步式表头生成流程

### Step 1：决定哪些列出现在虚拟表中（`active_cols`）

```python
# 扫描所有源文件的合计行，筛选"有数据"的列
for c in range(3, max_cols + 1):    # 跳过序号列(1)和姓名列(2)
    if 任一源文件该列合计值 != 0:
        active_cols.append(c)

virtual_cols = 2 + len(active_cols)  # 序号 + 结算单元名称 + 数据列
```

**规则**：从源文件列3开始，只要有一个源文件的合计行该列值非零，就纳入虚拟表。

### Step 2：从第1个源文件读取表头模板

从 `items[0]["path"]` 的第一个源文件中读取：

```python
hdr_rows[0] = {}  # 源文件行3 → 虚拟表第1层（列名）
hdr_rows[1] = {}  # 源文件行4 → 虚拟表第2层（子类别）
hdr_rows[2] = {}  # 源文件行5 → 虚拟表第3层（子明细）

src_merges = []   # 行3-5范围内所有合并单元格的 (r1, r2, c1, c2)
```

同时做**向右传播**处理：openpyxl 读取合并单元格时只有左上角有值，把值向右填满，确保"扣款明细"这类跨多列的标题每列都能读到。

### Step 3：写入并合并

#### 3a. 写入3行复合表头（行4-6）

```python
col_map = {}  # 源列号 → 虚拟列号
for vi, oc in enumerate(active_cols, 3):
    col_map[oc] = vi

# 输出布局（以30列标准白云文件为例）:
#  虚拟列 1  2    3      4       5         6         7     8~18    19   20    21
#  行4: 序号 | 单位名 | 基本工资 | 应发工资 | 单位五险 | 代理费 | 转账 | 扣款明细(11列) | 个税 | 实发 | 合计
#  行5:      |        |         |         |         |        |      | 养老|失业|...|合计|
#  行6:      |        |         |         |         |        |      | 单|个|单|个|...|
```

#### 3b. 动态合并（核心逻辑）

通过遍历第1个源文件的 `merged_cells.ranges`，将行3-5的合并规则复制到虚拟表：

```python
hmerged = set()  # 存 (virtual_row, virtual_col) 已合并的单元格

for (src_r1, src_r2, src_c1, src_c2) in src_merges:
    vc1 = col_map.get(src_c1)          # 源列 → 虚拟列
    vr1 = hdr_start_row + (src_r1 - 3) # 源行 → 虚拟行 (row3→0,row4→1,row5→2)
    
    if 单列纵向合并 (src_c1 == src_c2, src_r1 < src_r2):
        # 例: W4:W5 (工伤险), Z4:Z5 (单位代理费)
        → 虚拟表中跨 vr1→vr2 行合并该列
        
    elif 同行水平合并 (src_r1 == src_r2, src_c1 < src_c2):
        # 例: Q4:R4 (养老2列), AI3:AY3 (扣款明细17列)
        → 验证映射后列是否连续
        → 虚拟表中跨 mn→mx 列合并该行
```

**关键设计**：`hmerged` 存 `(row, col)` 元组而不是单纯列号。

| 场景 | 问题 | 解决方案 |
|------|------|---------|
| 行4"扣款明细"跨列8~18水平合并 | 列8~18全部标记"已合并" | 存 `(row, col)` → `{(4,8), (4,9), ...}` |
| 行5"养老"合并列8~9 | 检查 `(5,8)` 是否在 hmerged → 不在，可以合并 ✅ | 不同行的同列合并不冲突 |

#### 3c. 剩余列兜底合并

对还没有任何合并的列，如果行4+行5都为空值，自动做**跨3行纵向合并**（虚拟行4-6，即源行3-5）：

```python
for vi in range(1, virtual_cols + 1):
    if (hdr_start_row, vi) 到 (hdr_end_row, vi) 已有合并:
        continue
    if 行4该列为空 AND 行5该列为空:
        跨3行合并该列  # 如"序号""基本工资"等单值列头
```

## 三、列映射（`col_map`）详解

虚拟表的列布局：

| 虚拟列 | 内容 | 来源 |
|--------|------|------|
| 1 | 序号 | 固定 |
| 2 | 结算单元名称 | 固定（显示各源文件单位名） |
| 3+ | 数据列 | `active_cols` 依次映射 |

`col_map` 将源文件的列号映射到虚拟列号：

```python
col_map[source_col] = virtual_col  # source_col ∈ active_cols
col_map[17] = 8   # 例：扣款明细第一列 → 虚拟第8列
col_map[27] = 18  # 例：扣款合计列 → 虚拟第18列
```

**重要**：中间可能有列被 `active_cols` 过滤掉（如"交通补贴"全零），导致映射不连续：

```
源列: 7     9    10    11    12    17    18    19  ...  27    28    29    30
虚拟列: 3    4     5     6     7     8     9    10  ...  18   19    20    21
         ↑跳过8(交通补贴)    ↑跳过13-16(基数区域均为0)
```

这种不连续性会在合并映射时被 `col_map.get(c)` 返回 `None` 检查出来，水平合并要求中间的列全部映射且连续。

## 四、合并单元格类型与处理方式

源文件行3-5的合并单元格有 4 种，各自对应虚拟表中的处理：

| 类型 | 源文件例子 | 虚拟表处理 | 代码分支 |
|------|-----------|-----------|---------|
| **单列纵向**（行4→行5） | `W4:W5` (工伤险) | 虚拟行2→行3 纵向合并该列 | `src_c1 == src_c2` |
| **单列纵向**（行3→行5） | `A3:A5` (序号) | 被兜底合并覆盖（行4行5都空） | 兜底逻辑 |
| **同行水平**（行3独占） | `Q3:AA3` (扣款明细11列) | 虚拟行1水平合并对应列段 | `src_r1 == src_r2` |
| **同行水平**（行4独占） | `Q4:R4` (养老2列) | 虚拟行2水平合并对应列段 | `src_r1 == src_r2` |
| **三维合并**（跨行跨列） | `V3:W5` (交通补贴) | 视情况：行3-5+列22-23，部分列可能被过滤 | 拆解为水平和纵向 |

## 五、与之前硬编码方案的对比

| 维度 | 旧方案（已删除） | 新方案 |
|------|----------------|--------|
| 合并规则来源 | 硬编码 `[(17,18), (19,20), ...]` | 动态读取 `merged_cells.ranges` |
| 扣款明细块宽度 | 固定11列 | 自适应（11/12/13/17列均可） |
| 扣款明细起始列 | 固定列17 | 自适应（16/17/18/35均可） |
| 附加项支持 | 仅2项（代理费+扣款合计） | 自动支持任意附加项 |
| 合并冲突处理 | 按列号去重，不同行冲突 | 按 `(行,列)` 去重，不同行不冲突 |
| 映射连续性 | 假设列映射连续 | 检查连续性，不连续则跳过合并 |
| 其他格式（58列博众） | 不支持 | 理论上支持（`col_map` 映射即可） |

## 六、边界情况与注意事项

### 6.1 交通补贴跨2列（V3:W5）

博众58列格式中，"交通补贴"使用 `V3:W5` 三维合并（向右1格行+向下2格）。这种合并拆解后：
- 水平部分：行3列22-23 → 在虚拟表行1水平合并对应列
- 垂直部分：`V3:V5` 和 `W3:W5` → 在虚拟表被兜底逻辑覆盖（行4行5为空）

但如果列23被 `active_cols` 过滤掉而列22保留，则水平合并无法完整映射（不连续），该合并会被跳过，只保留兜底合并。

### 6.2 `active_cols` 过滤影响合并

如果某列的合计值为 0，该列不会出现在虚拟表中。这会导致：
- 部分合并单元格的映射列不连续 → 水平合并被跳过
- 被跳过的列享受兜底合并（行4行5都空 → 跨3行合并）
- 最终效果接近正确，但少了一些视觉合并

### 6.3 第1个源文件的选择

表头和合并规则来自第1个源文件。如果第1个文件的列数不是最大的（`max_cols`取所有文件最大值），`col_map` 中可能有部分列映射不到，这部分列的合并也会被跳过。

### 6.4 WPS COM 的 Merge 稳定性

`Range.Merge()` 在 WPS COM 中可能会对已合并的区域报错，因此使用了 `try/except` 包裹 + `hmerged` 检查双重防护。

```
读取所有 signed_ 文件
    ↓

    ↓

    ↓

    ↓

    ↓

    ↓

  列1=序号, 列2=结算单元名称, 列3+=active_cols映射
    ↓

  ├── 单列纵向合并 → 映射后合并虚拟表对应行范围

    ↓
WPS COM Paste 堆叠源文件原表
```
