## 一、标准表头结构（行1-5）

所有 `signed_*.xlsx` 文件的表头均为 **5行复合结构**，数据从第6行开始，末行为合计行。

### 行1：大标题（合并单元格 A1~AE1）

格式：`{单位名称}{年月}人才派遣人员工资发放表`

示例：
```
吉林大学东北亚学院2026年06月人才派遣人员工资发放表
```

### 行2：单位信息 + 填报时间

格式：`单位 | 名称：{单位名称} | 填报时间：{日期}`

> 注：未签字版本（非 `signed_`）格式有时为 `单位名称：{名称}`（无"单位 |"前缀）。

示例：
```
单位
  名称：吉林大学东北亚学院
  填报时间：2026-05-20
```

```

> **↕ = 向下合并1格（行4→行5贯通）**

4. **块锚点**：始终位于"基数区域"（社保/医保/工伤/公积金基数）之后、"个人所得税"之前

## 五、扣款明细块深度剖析（三级表头核心）

> 这是整个工资表头**最难、最核心**的部分。理解清楚这里，虚拟工资表合并表头就解决了一半。

### 5.1 三级表头结构

```
层级1 ─ 行3: |←                   扣  款  明  细                        →|  ← 整块水平合并
层级2 ─ 行4: |←养老→|←失业→|←医疗→|工伤|←公积金→|↕|↕|↕|...|↕|   ← 子类别（水平/纵向混合合并）
层级3 ─ 行5: |单|个|单|个|单|个|单|单|个|(空)..|              ← 单位/个人拆分（部分被覆盖）
```

### 5.2 块内列分组模式（所有变种统一规律）

无论扣款明细块总宽是11列、12列、13列还是17列，块内的列分组遵守固定模式：

#### 第1组：核心5项（恒占9列，永远不变）

| 块内位置 | 列数 | 行4内容 | 合并方式 | 行5内容 | 对应社保项目 |
|---------|------|---------|---------|---------|------------|
| 第1~2列 | 2 | "养老" | **向右合并1格**(H2) | 第1列="单位", 第2列="个人" | 养老保险 |
| 第3~4列 | 2 | "失业" | **向右合并1格**(H2) | 第3列="单位", 第4列="个人" | 失业保险 |
| 第5~6列 | 2 | "医疗" | **向右合并1格**(H2) | 第5列="单位", 第6列="个人" | 医疗保险 |
| 第7列 | 1 | "工伤险" | **向下合并1格**(V4→5) | 第7列="单位"(被覆盖) | 工伤保险 |
| 第8~9列 | 2 | "公积金" | **向右合并1格**(H2) | 第8列="单位", 第9列="个人" | 住房公积金 |

**核心5项合并规律总结：**
```
行4: |← 养老 H2 →|← 失业 H2 →|← 医疗 H2 →|工伤 V|← 公积金 H2 →|
行5: |  单位  |  个人  |  单位  |  个人  |  单位  |  个人  |  单位 |  单位  |  个人  |
```

> H2 = 向右合并1格（2列宽，仅行4有效）  
> V = 向下合并1格（行4→行5贯通）

#### 第2组：附加项（0~N列，每项1列纵向贯通）

从块内第10列开始，每列为一个独立附加项：

| 附加项 | 合并方式 | 行5状态 |
|-------|---------|--------|
| 单位代理费 | **向下合并1格**(V4→5) | 被行4覆盖 |
| 扣款合计 | **向下合并1格**(V4→5) | 被行4覆盖 |
| 大病险（单位） | **向下合并1格**(V4→5) | 被行4覆盖 |
| 大病险合计 | **向下合并1格**(V4→5) | 被行4覆盖 |
| 公务员医疗补助 | **向下合并1格**(V4→5) | 被行4覆盖 |
| 雇主责任险 | **向下合并1格**(V4→5) | 被行4覆盖 |
| 补缴及退款保险差额（单位） | **向下合并1格**(V4→5) | 被行4覆盖 |
| 补缴及退款保险差额（个人） | **向下合并1格**(V4→5) | 被行4覆盖 |
| 意外险 | **向下合并1格**(V4→5) | 被行4覆盖 |
| 返餐费 | **向下合并1格**(V4→5) | 被行4覆盖 |
| ... | ... | ... |

> 所有附加项的合并模式统一：行4有名称，行5无内容（被行4向下合并覆盖）。

#### 统一规则

| 规则 | 描述 |
|------|------|
| **R1** | 前9列（5组）是**核心项**，所有工资表必有 |
| **R2** | 第10列起是**附加项**，数量可变（0~N），每项1列 |
| **R3** | 核心项的**列宽固定**（2,2,2,1,2），附加项每项1列 |
| **R4** | 核心项合并模式固定（养老H2, 失业H2, 医疗H2, 工伤V, 公积金H2） |
| **R5** | 附加项合并模式统一（行4单列V向下合并至行5） |
| **R6** | 行5第10列起全部为 `None`（被行4覆盖） |
| **R7** | 行3整块水平合并，合并宽度 = 9 + 附加项数 |

### 5.3 所有发现变种的扣款明细块一览

| 变种 | 总宽 | 基础9列 | 附加项 | 出现于 |
|------|------|---------|-------|--------|

### 5.4 生成虚拟合并表头时的策略

基于上述分析，合并工资表时构建统一表头的**推荐策略**：

1. **扫描所有源文件**，收集每个文件的扣款明细行4内容（附加项名称列表）
2. **取并集**：将所有出现的附加项去重合并（保持列6养老失业医疗工伤公积金固定）
3. **构建统一表头**：
   - 行3：整块水平合并，宽度 = 9 + 去重后附加项数
   - 行4：9列核心（固定合并模式）+ 各附加项依次排列（每项1列，V合并）
   - 行5：9列核心子列名 + 其余空
4. **数据填充**：按统一表头逐源文件填入数据，缺失的附加项列填0或空

这样无论输入文件是29列还是58列，都能生成结构一致的合并输出。

- **数据开始行**：第6行
- **数据结束行**：不计合计行的话，为 `max_row - 1`
- **合计行**：末行，第1列内容含"合计"关键字
- **签字图片**：在 `signed_` 版本中，图片悬浮在表格区域外或右侧（通过 win32com 复制时需保留）

## 六、与汇总表（验证明细）的关系

合并时创建的"虚拟表"结构（见 `merge_payrolls_simple` 代码）：

```
行1: 大标题（单位名 + 工资表合集 + 年月范围）
行2: 单位名（左）| 打印时间（右）
行3+: 虚拟列头 + 各源文件整表堆叠
```

虚拟表**不直接复用源文件的5行复合头**，而是：
1. 扫描第1个源文件的行3-5作为列头参考
2. 扫描所有源文件的合计行来筛选"有数据列"
3. 构建简化后的虚拟列头：序号 + 结算单元名称 + 数据列

非标准 `signed_` 文件（如 `(吉林大学）深部探测与成像全国重点实验室2026年06月工资表.xlsx`）结构与 `signed_` 一致但列数可能更少，不应作为合并输入。

## 八、注意事项

1. **列数不对齐的兼容**：29/30/31/33列变种中，扣款明细~末尾的列位置会偏移，不能硬编码列号映射
2. **身份证科学记数法**：读身份证列时应强制文本格式，避免 `1.23E+17` 问题
3. **合计行检测**：通过第1列包含"合计"字符串来定位，而非固定行号
4. **合并单元格读值**：`openpyxl` 读合并区域时，只有左上角单元格有值，其余为 `None`
5. **`signed_` vs 未签字**：未签字文件有时格式有微小差异（行2格式不同、列数更少），应以 `signed_` 为准

### 9.5 实际数据布局说明

列7~23（16个工资项）并不全是加法项：
- **正向项**（加）：补发工资、奖金、岗位工资、绩效奖金、一次性奖金、津贴、夜餐费、加班费、补贴、交通补贴

- 应发工资（列24）= 上述各项的代数和

### 9.6 与白云标准的通用合并模式对比

```

  行3: 大多单列直接跨行3-5(无合并) | ←扣款明细11列→
  行4: 仅扣款块有子头              |←养老→|←失业→|←医疗→|工伤↕|←公积金→|代理↕|合计↕
  行5:                              |单|个|单|个|单|个|单 |单|个 |(空)|(空)

博众（58列）：
  行3: ↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓↓ |←扣款明细17列→| ↓↓↓↓↓↓↓
  行4: (全被行3向下合并覆盖，仅有扣款块有内容) |←养→|←失→|←医→|工|←公→|↕|↕|↕|↕|↕|↕|↕|

  ↓ = 向下合并2格（行3→行5）

### 9.7 对合并代码的启示

1. **列数差异巨大**（58 vs 30），不能假设固定列数
2. **扣款明细块宽度不同**（17 vs 11），不能硬编码块内偏移
3. **行3几乎全列合并**（不像白云仅扣款块），openpyxl 读到空值不一定是真空
4. **合计行不在末行**（第902行，之后还有备注、签字行）
5. **有签字行**需跳过
6. **列23被列22合并覆盖**——遍历列头时要用 `merged_cells.ranges` 检查
7. **对于汇总虚拟表**——两种格式合并到一个表时，需统一为通用接口
